
There is a scaling issue, which is reproducible with 64 cores (128 logical processors) on:
1. TCEC game play of Bagatur (https://tcec-chess.com/)
1. CentOS Stream release 8 and Java OpenJDK-1.8.0-x64 on a Google Cloud Platform (GCP) VM
2. Windows Server and Java8-u361-x64 on GCP VM

The issue persists, although there is no threads synchronization at all.
All threads are reading and writing in the same two 1 dimensional arrays of longs (keys and values).
I tried some ideas to solve the issue, currently without success. For example here are some directions in which I was trying to make the fix:
1. The distribution of hashkeys is good enough, they are almost equally distributed, so it is NOT like many threads are reading/writing the the very close memory segments more often.
2. Skipping TT access (reads/writes) on shallow depth (up to depth 5). This solves the scaling issue but the ELO strength is decreased as well as the depth, which search achieves for a given time.
3. Try Java specific anti-patterns just to solve it, again without success (e.g. static arrays instead of class member references). 
My research shows that the root cause is in the way the Java handles the data transfer between the main memory and the CPUs L1/L2/L3 caches.
Maybe this particular use case, where there is two big (up to 16 GB) arrays of longs, and without any synchronization many threads are accessing the data inside them, is never considered and tested from the Java creators/teams.

As for now, I have introduced the CountTranspositionTables UCI option, which is set to 1 by default.
With 64 cores, the results below show there is also balanced performance with 8 Transposition Tables, and this UCI option makes it possible one to experiment.
